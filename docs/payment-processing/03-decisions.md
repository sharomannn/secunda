# Архитектурные решения: Payment Processing Microservice

## ADR-01: Использование Outbox Pattern для гарантии доставки событий

**Статус:** Принято

**Контекст:**
При создании платежа необходимо гарантировать, что событие будет опубликовано в RabbitMQ. Прямая публикация в брокер после записи в БД создаёт риск потери события при сбое между двумя операциями (dual-write problem).

**Решение:**
Использовать Outbox Pattern:
1. Записывать Payment и Outbox-событие в одной транзакции БД
2. Отдельный процесс читает неопубликованные события из таблицы `outbox` и публикует их в RabbitMQ
3. После успешной публикации обновлять статус события на `published`

**Обоснование:**
- Гарантирует at-least-once delivery событий
- Транзакционная консистентность между Payment и Outbox
- Устойчивость к сбоям RabbitMQ — события накапливаются в БД и публикуются при восстановлении
- Стандартный паттерн для микросервисной архитектуры

**Последствия:**
- Дополнительная таблица `outbox` в БД
- Фоновый процесс для публикации событий (можно реализовать через FastAPI background tasks или отдельный процесс)
- Небольшая задержка между созданием платежа и публикацией события (приемлемо для асинхронной обработки)
- Необходимость периодической очистки старых опубликованных событий

**Альтернативы:**
- Прямая публикация в RabbitMQ — отклонено из-за риска потери событий
- Transactional Outbox с CDC (Change Data Capture) — избыточно для текущих требований

## ADR-02: Idempotency Key для защиты от дублей

**Статус:** Принято

**Контекст:**
Клиент может повторно отправить запрос на создание платежа из-за сетевых проблем или таймаутов. Необходимо предотвратить создание дубликатов платежей.

**Решение:**
Требовать обязательный заголовок `Idempotency-Key` в запросе POST /api/v1/payments:
1. Проверять наличие платежа с таким ключом перед созданием
2. Если платёж существует — возвращать его (200 OK вместо 202 Accepted)
3. Если платёж не существует — создавать новый с этим ключом
4. Уникальный индекс на поле `idempotency_key` в БД

**Обоснование:**
- Стандартный подход для идемпотентных API (Stripe, PayPal используют аналогично)
- Защита от дублей на уровне приложения
- Простота реализации и понимания
- Клиент контролирует уникальность ключа

**Последствия:**
- Клиент обязан генерировать уникальные ключи (UUID рекомендуется)
- Необходимость валидации наличия заголовка (422 если отсутствует)
- Уникальный индекс в БД на `idempotency_key`
- Политика хранения ключей (рекомендуется хранить бессрочно или с TTL 24-48 часов)

**Альтернативы:**
- Автоматическая дедупликация по полям запроса — отклонено, т.к. клиент может намеренно создавать идентичные платежи
- Опциональный Idempotency-Key — отклонено, т.к. не гарантирует защиту

## ADR-03: Статический API Key для аутентификации

**Статус:** Принято

**Контекст:**
Необходима простая аутентификация для защиты API от несанкционированного доступа. Требования ТЗ указывают на статический API ключ.

**Решение:**
Использовать статический API ключ в заголовке `X-API-Key`:
1. Ключ хранится в переменной окружения `API_KEY`
2. Middleware проверяет наличие и корректность ключа для всех эндпоинтов
3. Возврат 401 Unauthorized при отсутствии или неверном ключе

**Обоснование:**
- Соответствует требованиям ТЗ
- Простота реализации и использования
- Достаточно для тестового задания и внутренних сервисов
- Не требует инфраструктуры для управления токенами

**Последствия:**
- Один ключ для всех клиентов (нет разграничения доступа)
- Необходимость ротации ключа при компрометации (требует перезапуска сервиса)
- Ключ должен передаваться по HTTPS в production

**Альтернативы:**
- OAuth 2.0 / JWT — избыточно для текущих требований
- Множественные API ключи с разграничением — не требуется по ТЗ
- mTLS — сложность настройки превышает требования

**Рекомендации для production:**
- Использовать JWT с коротким TTL
- Реализовать rate limiting
- Логировать все попытки аутентификации

## ADR-04: Retry с экспоненциальной задержкой для webhook

**Статус:** Принято

**Контекст:**
Webhook-эндпоинт клиента может быть временно недоступен. Необходимо повторить отправку с разумными интервалами, не перегружая систему клиента.

**Решение:**
Реализовать retry-логику с экспоненциальной задержкой:
- Количество попыток: 3
- Задержки: 1 секунда, 2 секунды, 4 секунды
- После исчерпания попыток — отправка сообщения в DLQ

**Обоснование:**
- Экспоненциальная задержка даёт время на восстановление сервиса клиента
- 3 попытки — баланс между надёжностью и скоростью обработки
- Общее время retry: ~7 секунд (приемлемо для асинхронной обработки)
- Стандартный паттерн для распределённых систем

**Последствия:**
- Задержка доставки webhook до 7 секунд при проблемах
- Необходимость логирования всех попыток для отладки
- DLQ для ручного анализа невосстановимых ошибок

**Альтернативы:**
- Фиксированная задержка — отклонено, может перегрузить восстанавливающийся сервис
- Больше попыток (5-10) — отклонено, увеличивает время обработки
- Бесконечные retry — отклонено, блокирует очередь

## ADR-05: Dead Letter Queue для невосстановимых ошибок

**Статус:** Принято

**Контекст:**
После исчерпания всех попыток обработки сообщение должно быть сохранено для анализа, а не потеряно.

**Решение:**
Использовать Dead Letter Queue (DLQ):
1. Настроить DLQ `payments.new.dlq` в RabbitMQ
2. После 3 неудачных попыток обработки сообщение автоматически перемещается в DLQ
3. DLQ доступна для ручного анализа и повторной обработки

**Обоснование:**
- Предотвращает потерю сообщений
- Позволяет анализировать причины ошибок
- Не блокирует основную очередь
- Стандартная практика для RabbitMQ

**Последствия:**
- Необходимость мониторинга размера DLQ
- Процесс для ручной обработки сообщений из DLQ
- Дополнительная очередь в RabbitMQ

**Альтернативы:**
- Логирование и удаление — отклонено, теряется возможность повторной обработки
- Бесконечные retry — отклонено, блокирует очередь

## ADR-06: SQLAlchemy 2.0 в асинхронном режиме

**Статус:** Принято

**Контекст:**
FastAPI поддерживает асинхронные обработчики. Необходимо выбрать ORM для работы с PostgreSQL.

**Решение:**
Использовать SQLAlchemy 2.0 с асинхронным драйвером (asyncpg):
- Асинхронные сессии через `AsyncSession`
- Декларативные модели с `DeclarativeBase`
- Async engine для подключения к БД

**Обоснование:**
- Соответствует требованиям ТЗ
- Полная поддержка async/await в FastAPI
- SQLAlchemy 2.0 — современный стандарт с улучшенной типизацией
- asyncpg — высокопроизводительный драйвер для PostgreSQL

**Последствия:**
- Все ORM-операции должны использовать `await`
- Необходимость понимания async context managers
- Миграции через Alembic (поддерживает async)

**Альтернативы:**
- Синхронный SQLAlchemy — отклонено, блокирует event loop
- Tortoise ORM — отклонено, меньше зрелости и экосистемы
- Прямые SQL-запросы — отклонено, теряется абстракция ORM

## ADR-07: FastStream для работы с RabbitMQ

**Статус:** Принято

**Контекст:**
Требуется библиотека для работы с RabbitMQ в асинхронном режиме. ТЗ указывает FastStream.

**Решение:**
Использовать FastStream для:
- Публикации событий в RabbitMQ
- Consumer для обработки сообщений из очереди
- Настройки retry и DLQ

**Обоснование:**
- Соответствует требованиям ТЗ
- Современная библиотека с поддержкой async/await
- Декларативный стиль (похож на FastAPI)
- Встроенная поддержка retry и DLQ

**Последствия:**
- Зависимость от относительно новой библиотеки
- Необходимость изучения FastStream API

**Альтернативы:**
- aio-pika — отклонено, более низкоуровневая библиотека
- Celery — отклонено, избыточная функциональность для текущих требований

## ADR-08: UUID для идентификаторов платежей

**Статус:** Принято

**Контекст:**
Необходимо выбрать тип идентификатора для платежей.

**Решение:**
Использовать UUID (версия 4) для поля `id` в таблице `payments`:
- Генерация на уровне приложения
- Тип `UUID` в PostgreSQL
- Возврат в API как строка

**Обоснование:**
- Глобальная уникальность без координации с БД
- Безопасность — невозможно угадать ID других платежей
- Подходит для распределённых систем
- Стандарт для публичных API

**Последствия:**
- Больший размер по сравнению с integer (16 байт vs 4-8 байт)
- Невозможность сортировки по времени создания (использовать `created_at`)
- Индексы занимают больше места

**Альтернативы:**
- Auto-increment integer — отклонено, предсказуемость ID — риск безопасности
- ULID — отклонено, избыточно для текущих требований

## ADR-09: Decimal для хранения сумм платежей

**Статус:** Принято

**Контекст:**
Необходимо точно хранить денежные суммы без ошибок округления.

**Решение:**
Использовать тип `NUMERIC(precision=10, scale=2)` в PostgreSQL:
- Precision: 10 (максимум 8 цифр до запятой)
- Scale: 2 (2 цифры после запятой)
- В Python: `Decimal` из модуля `decimal`

**Обоснование:**
- Точное представление денежных сумм (без ошибок float)
- Стандарт для финансовых приложений
- Поддержка арифметических операций без потери точности

**Последствия:**
- Необходимость использовать `Decimal` в Python (не `float`)
- Валидация на уровне Pydantic: `condecimal(gt=0, decimal_places=2)`

**Альтернативы:**
- Float — отклонено, ошибки округления недопустимы для денег
- Integer (копейки) — отклонено, усложняет API и логику

## ADR-10: Enum для статусов и валют

**Статус:** Принято

**Контекст:**
Статус платежа и валюта имеют фиксированный набор значений.

**Решение:**
Использовать Python Enum и PostgreSQL ENUM:
- `PaymentStatus`: pending, succeeded, failed
- `Currency`: RUB, USD, EUR

**Обоснование:**
- Типобезопасность на уровне приложения и БД
- Валидация на уровне БД (невозможно вставить некорректное значение)
- Автодополнение в IDE
- Явная документация допустимых значений

**Последствия:**
- Изменение набора значений требует миграции БД
- Необходимость синхронизации Enum между Python и PostgreSQL

**Альтернативы:**
- VARCHAR без ограничений — отклонено, нет валидации на уровне БД
- CHECK constraint — отклонено, менее явно чем ENUM
